## 总览
本项目采用 **FastAPI + Jinja（SSR）+ Tailwind** 的轻量架构，实现“万能视频下载”的核心业务流程，下载能力统一封装 `yt-dlp`（不自研解析）。

### 设计目标
- **快速可用**：本地/公网一套代码
- **可扩展**：后续叠加 formats、字幕、总结、付费等能力
- **风险可控**：cookies 默认关闭、临时存储、限流与自动清理

## UI 设计（对齐参考站）
参考 `https://ai.codefather.cn/painting` 的关键风格点迁移到“下载”场景：
- **Hero 大标题**：黑色主标题 + 蓝色强调词（吸引付费）
- **胶囊输入条**：圆角 pill + 图标 + 蓝色按钮
- **卡片瀑布流**：大圆角、柔和阴影、留白充足、标签 chips
- **极简顶部栏**：汉堡菜单/Logo + Pro 入口

实现位置：
- 前端模板：`backend/app/templates/layout.html`、`backend/app/templates/index.html`
- 静态脚本：`backend/app/static/app.js`

## 后端架构
### 组件划分
- `backend/app/main.py`：API + SSR 页面 + SSE
- `backend/app/jobs.py`：内存 JobStore、事件队列、TTL 清理、任务目录
- `backend/app/ydl.py`：`yt-dlp` 封装（extract/download/progress hook）
- `backend/app/settings.py`：环境变量配置

### 核心数据结构
- **Job**：一条链接对应一个 Job（id、url、status、output_path、expires_at、ip）
- **Progress**：percent/speed/eta/filename/message
- **Status**：queued/downloading/finished/failed/expired

### API 列表
- `GET /`：首页（SSR）
- `POST /api/jobs`：创建任务（支持多行 URL 批量）
- `GET /api/jobs/{id}`：查询任务状态
- `GET /api/jobs/{id}/events`：SSE 推送进度/状态事件
- `GET /api/jobs/{id}/file`：下载成品文件

## 任务执行模型
- 创建任务后，后台启动 `asyncio.create_task`，内部用 `ThreadPoolExecutor` 跑阻塞的 `yt-dlp` 下载
- `progress_hooks` 在下载线程里触发，通过 `loop.call_soon_threadsafe` 回传到事件循环更新进度
- SSE 订阅端持续读取该 job 的事件队列并推送给前端

## cookies 能力（默认关闭）
- 配置：`FVD_ENABLE_COOKIES=false`（默认）
- 开启时需要：`FVD_ADMIN_TOKEN`，并在请求头里携带 `X-Admin-Token`
- cookies 文件仅用于本次任务，落盘到 job 临时目录，并随 TTL 清理自动删除

## 临时存储与清理
- 目录：`/tmp/fvd/{job_id}/...`
- TTL：`FVD_TTL_SECONDS`（默认 3600）
- 清理：后台定时任务 `FVD_CLEANUP_INTERVAL_SECONDS`（默认 300）扫描过期 job 并删除目录

## 运行模式
### 真实下载模式
- `FVD_DEMO_MODE=false`（默认）：走 `yt-dlp` 真实下载

### 演示/CI 模式（无外网也能跑通流程）
- `FVD_DEMO_MODE=true`：不访问网络，模拟进度并生成占位文件，确保端到端链路可验证

## 关键配置（环境变量）
- `FVD_DEMO_MODE`：true/false
- `FVD_ENABLE_COOKIES`：true/false
- `FVD_ADMIN_TOKEN`：管理员 token
- `FVD_TTL_SECONDS`：临时文件保留时长
- `FVD_MAX_URLS_PER_REQUEST`：单次最多 URL 数
- `FVD_MAX_ACTIVE_JOBS_PER_IP`：单 IP 同时进行任务数

## AI 视频总结（P0 实施版）
### 目标与边界
- 目标：在现有下载能力上增加“下载后自动总结”基础能力，帮助快速理解长视频内容。
- 平台边界：P0 仅对 **B站链接** 启用总结流程；其他平台默认跳过总结。
- 模型：统一使用 **DeepSeek**（默认 `deepseek-chat`）。
- 转写策略：**优先抓取 B站字幕**；字幕不可用时回退到本地开源 ASR（Whisper）。

### 处理流程
1. 创建下载任务（可批量），为每个 Job 标记 `enable_summary`。
2. 下载完成后进入总结流水线：
   - `transcribing`：先尝试 `yt-dlp` 提取字幕轨道并拉取字幕文本；
   - 若失败，提取音频（ffmpeg）并使用 Whisper 本地转写；
   - `summarizing`：调用 DeepSeek 生成结构化总结。
3. 输出落盘：
   - `transcript.json`：转写文本与来源信息（subtitle/asr）；
   - `summary.json`：结构化总结结果。
4. SSE 推送总结事件，前端可在任务卡中查看总结。

### 数据结构扩展
- Job 增加字段：
  - `summary_status`：`not_requested | queued | transcribing | summarizing | done | failed | skipped`
  - `summary_error`：总结失败原因
  - `summary_result`：结构化总结结果
  - `meta.enable_summary`：该任务是否启用总结

### API 扩展
- `GET /api/jobs/{id}`：返回 `summary_status/summary_error/summary_result`
- `GET /api/jobs/{id}/summary`：获取总结详情（done 后可读）
- `POST /api/jobs/{id}/summary/retry`：重试总结（不重复下载）
- `GET /api/jobs/{id}/events`：新增 `summary` 事件类型

### 结构化输出（P0）
总结结果按 JSON 输出，包含：
- `outline`：视频大纲（3-8条）
- `key_points`：核心知识点（5-12条）
- `action_items`：可执行建议（3-8条）
- `terms`：关键词术语（5-15条）
- `model`：模型名称

### 运行与配置
- `FVD_ENABLE_AI_SUMMARY=true|false`：是否启用 AI 总结
- `FVD_SUMMARY_ONLY_BILIBILI=true|false`：是否仅 B站启用总结
- `FVD_DEEPSEEK_API_KEY`：DeepSeek API Key（生产必填）
- `FVD_DEEPSEEK_BASE_URL`：DeepSeek API Base URL（默认 `https://api.deepseek.com`）
- `FVD_DEEPSEEK_MODEL`：模型名（默认 `deepseek-chat`）
- `FVD_ASR_MODEL_NAME`：Whisper 模型名（默认 `base`）

### 失败与降级策略
- 下载成功但总结失败：任务保留下载成功状态，`summary_status=failed`，支持重试总结。
- 总结能力关闭或非 B站：`summary_status=skipped`。
- `DEMO_MODE` 下：生成演示转写与演示总结，保障 CI 和无外网环境可验证。
